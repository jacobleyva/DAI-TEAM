# Beliefs

> **Strong positions the team operates from.** These are the frames behind every design decision. They are explicit so they can be argued with — and so a new teammate (human or AI) can see them. Beliefs change when they get refuted, not when they get uncomfortable.

## How to write a belief

- **One sentence per belief.** If you need a paragraph, you have two beliefs.
- **Falsifiable.** A good belief can lose. "We value quality" is too soft. "Code without tests does not ship" is a belief.
- **Operate-able.** A belief should change how decisions are made. If it has no behavioral consequence, it is decoration.
- **Owned.** When a belief is refuted by evidence, capture the refutation here with the date — do not silently delete it. Retired beliefs are leverage.

## Active beliefs

(Replace the `(SEED)` entries below.)

- **B1:** (SEED) Scaffolding beats model. A weaker model with stronger scaffolding outperforms a stronger model with weaker scaffolding on production work. This is why this workspace exists.
- **B2:** (SEED) Discipline lives in the file, not in the head. Anything that depends on a teammate remembering a rule is a rule that will be broken. Put it in `core/`.
- **B3:** (SEED) Reversible beats clever. The smallest reversible step that reduces uncertainty is almost always the right next move when uncertainty is high.
- **B4:** (SEED) Verification is the difference between "done" and "claimed done". An ISC without a tool probe is a lie waiting to happen.
- **B5:** (SEED) Different AI vendors have different design pressures. Claude Code's DAI can be light because the model carries discipline; Codex's DAI must be deterministic because the model won't. Both reach the same operational floor by different routes.
