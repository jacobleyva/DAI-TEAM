# How This Workspace Works

## The Model

This workspace separates work into a few stable layers:

- `core/` for rules
- `team/` for collaboration
- `memory/` for durable shared state
- `projects/` for active work
- `knowledge/` for reference material
- `templates/` for reusable formats
- `skills/` for repeatable behaviors
- `tools/` for helper scripts

## Why That Matters

- it keeps the repo from becoming one giant note
- it makes context easier to find
- it gives each session a predictable start
- it lets multiple people work without overwriting each other
- it creates a shared operating model instead of a pile of ad hoc files

## What To Do With New Information

- if it affects everyone, put it in shared docs
- if it is project-specific, put it in `projects/`
- if it is durable background, put it in `memory/`
- if it is reusable format or method, put it in `templates/` or `skills/`
- if it is just local scratch, keep it out of shared surfaces
